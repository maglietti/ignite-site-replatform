        <dl className={styles.definitionBox}>
          <dt>Write-through and write-behind capabilities</dt>
          <dd>
            Ignite can automatically write-through or write-behind all the changes to an external database
            for every key-value request issued by your applications.
          </dd>
        </dl>

        <dl className={styles.definitionBox}>
          <dt>Read-through capabilities</dt>
          <dd>
            Read-through means that Ignite can automatically load data from an external database if a record
            is missing in memory.
          </dd>
        </dl>
      </div>
      <div className={styles.featureImage}>

        <div className={styles.imageCaption}>
          <p>1. Call to the near cache</p>
          <p>2. Call to the Ignite cluster <span>(if a record is not found in the near cache)</span></p>
          <p>3. Read-through from the external database <span>(if the record is not in the Ignite cluster yet)</span></p>
        </div>
      </div>
      <div className={styles.featureText}>
        <h3>Near Cache To Process Your Requests In Microseconds</h3>
        <p>
          A near cache is a local client-side cache that stores the most frequently used data on the
          application end.
        </p>

        <h4>How it works</h4>
        <p>
          It is applied to applications that require a consistent response time of a few <strong>microseconds.</strong> Ignite
          automatically invalidates and updates the near cache.
        </p>
        <p>
          Whenever the primary copy of a record gets updated on a server node, Ignite propagates the change
          to all the nodes that store the record's backup copy, as well as to the applications that keep
          the record's copy in their near caches.
        </p>
      </div>
    </div>
  </div>
</section>
